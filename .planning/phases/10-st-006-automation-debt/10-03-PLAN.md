    **When to Use:** Trigger phrases: "is my team accumulating automation debt?", "are we shifting the burden in our AI pipeline?", "detect automation debt in our tool stack", "which fixes are symptomatic vs fundamental?", "automation debt audit", "are our quick fixes eroding our capacity?"

    **5-step workflow:**
    1. **Describe the System** -- Ask the user to describe the AI system experiencing a recurring problem. Capture: system name, recurring symptom, what keeps breaking.
    2. **Inventory Automation Layers** -- List all fixes, workarounds, and solutions applied. For each: name, type (quick_fix/fundamental/transition), side effects, how long active, times applied. Guide user to distinguish symptomatic vs fundamental.
    3. **Identify the Fundamental Solution** -- Ask: "What would truly fix the root cause?" If user says "I don't know" -- that's a signal of Knowledge Drain. Capture fundamental solution description and viability ratings (now vs original).
    4. **Run Detection** -- Call detect_automation_debt MCP tool with the gathered AutomationLayers and fundamental solution. Format input as: system_description, automation_layers list, fundamental_solution, viability ratings.
    5. **Interpret Results** -- Walk through the output: debt_score (explain severity), erosion_channels (which are active and why), automation_coverage_estimate (how much is fundamental vs symptomatic), recommendations (actionable next steps). If debt_score > 60, flag as Critical and recommend immediate transition strategy.

    **MCP Tool Integration:**
    - Primary tool: detect_automation_debt
    - Input format: DetectAutomationDebtInput with system_description, automation_layers (list of AutomationLayer), fundamental_solution, fundamental_viability_now, fundamental_viability_original
    - Output interpretation: debt_score (0-33 Healthy, 34-66 At Risk, 67-100 Critical), erosion_channels (list with active/inactive), automation_coverage_estimate (% non-symptomatic), recommendations

    **3 examples:**
    1. LLM validation stack: 3 quick_fix layers (retry logic, output validator, human review), 0 fundamental. Debt score: 78 (Critical). Active channels: Knowledge Drain, Complexity Creep.
    2. Agent error handling: 2 quick_fix (exception handlers, fallback paths) + 1 transition (sub-agent prototype). Debt score: 45 (At Risk). Active channel: Complexity Creep.
    3. Tool latency: 1 quick_fix (cache layer) + 1 fundamental (pipeline optimization, 3 months active). Debt score: 22 (Healthy). No active erosion channels.

    **Troubleshooting:**
    - "All my fixes classify as symptomatic" -- Check if any address root cause. If genuinely all symptomatic, that's a valid finding (high debt score expected).
    - "Debt score seems too low" -- Ensure times_applied and duration_active are filled in. These increase erosion signals.
    - "I can't identify the fundamental solution" -- This is itself a sign of Knowledge Drain erosion. Note it in the analysis.
    - "Should I include successful fixes?" -- Yes, include all fixes. Even successful ones may be symptomatic.

    **Reference card (automation-debt-reference.md, ~1,000 chars):**
    - 3 layer types: quick_fix (treats symptom), fundamental (addresses root cause), transition (maintains fix while building fundamental)
    - 3 erosion channels: Knowledge Drain (fixes prevent learning), Complexity Creep (workarounds add layers), Normalization (team accepts degraded state)
    - Debt score tiers: Healthy (0-33), At Risk (34-66), Critical (67-100)
    - Quick diagnostic: "How many of your fixes address the symptom vs the root cause? If >60% symptomatic, check for burden shifting."
    - B1/B2/R1 reminder: B1 = symptomatic loop (balancing), B2 = fundamental loop (balancing, delayed), R1 = erosion loop (reinforcing)
