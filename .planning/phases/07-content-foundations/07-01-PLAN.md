Document structure:

## Agent Experience Checkpoint

### Purpose
Define the minimum conceptual knowledge students need before entering ST-004 (Agent Feedback Loops). This is NOT a gate -- it is a warm-up that activates prior knowledge and flags gaps.

### Entry Requirements (Conceptual Only)
- Can explain what an AI agent does that a chatbot does not (agency, tool use, planning)
- Can describe what a "tool call" means in an agent context
- Understands why an agent might need memory (context persistence across turns)
- Knows that agents can fail in loops (retry, escalation, cost spirals)
- No specific framework knowledge required (no LangChain, CrewAI, AutoGen specifics)

### 3-Question Self-Assessment Checklist
Place at top of ST-004 cubelet markdown. Casual tone, warm-up not a test.

1. "Can you explain what an AI agent does that a chatbot doesn't?" (Expected: mentions autonomy, tool use, or multi-step planning)
2. "What's a tool call?" (Expected: agent invoking an external capability -- API, database, function)
3. "Why might an agent need memory?" (Expected: maintaining context, learning from past interactions, avoiding repeated mistakes)

Format: Simple checkbox list. No scoring. If student checks all 3, proceed. If not, primer panel provides the background.

### Primer Panel Specifications
All 3 new cubelets get always-accessible collapsible sidebar primer panels in their artifacts:

**ST-004 Primer: Agent Basics**
- What is an AI agent (vs chatbot): autonomy, tool use, planning loops
- What are tools: external capabilities an agent can invoke (APIs, databases, functions)
- What is agent memory: context that persists across interactions
- What are evaluators: components that judge output quality
- What are constraints: rules limiting agent behavior (budget, time, safety)
- Cross-reference: links to ST-001 concepts (loop polarity, reinforcing vs balancing)

**ST-005 Primer: Tool Orchestration Basics**
- What is a tool dependency: one tool requires another's output
- What is redundancy: multiple tools doing the same thing
- What is blast radius: how many things break when one tool fails
- Cross-reference: links to ST-002 concepts (Meadows hierarchy, leverage scoring)

**ST-006 Primer: Automation Debt Basics**
- What is a quick fix in automation: temporary workaround that reduces immediate pain
- What is a fundamental solution: root-cause fix that eliminates the problem permanently
- What is erosion: how quick fixes make fundamental solutions harder over time
- Cross-reference: links to ST-003 concepts (B1/B2/R1 loops, shifting the burden)

### Belt and Suspenders
Agent vocabulary is ALSO woven into cubelet WHAT and HOW faces, not just primer panels. Students encounter definitions in the cubelet content itself AND have the primer as a reference.

### Primer Panel Cross-References
Each primer panel includes direct links to prerequisite cubelet tabs in the preview app:
- ST-004 primer links to ST-001 tab
- ST-005 primer links to ST-002 tab
- ST-006 primer links to ST-003 tab

Create a data model specification document with field names, types, validation rules, and example payloads for all 5 new Pydantic models. Per user decision: spec depth is field names + types + example values, not full Pydantic code. Models compose with existing CausalLink and FixRecord.

Document structure:

## Data Model Specifications

### Design Principles
- All new models compose with existing models (no duplication of proven logic)
- Follow existing patterns: ConfigDict(str_strip_whitespace=True), Field with description/constraints
- Follow existing ResponseFormat enum pattern for output formatting
- All models will be added to cubelets_mcp_server.py alongside existing models

### Model 1: AgentComponent

Purpose: Represents a node in an agent architecture diagram (used by ST-004 and ST-005)

Fields:
- id: str (unique identifier, auto-generated UUID if not provided)
- name: str (display name, e.g. "GPT-4 Router Agent", min 1, max 200)
- component_type: Literal["agent", "tool", "memory", "evaluator", "constraint"] (fixed 5 types per user decision)
- description: str (optional, what this component does, max 500)
- position_x: Optional[float] (canvas x position, for layout)
- position_y: Optional[float] (canvas y position, for layout)

Validation rules:
- component_type must be one of the 5 fixed types
- name must be non-empty after whitespace stripping

Example payload:
```json
{
  "id": "agent-1",
  "name": "GPT-4 Router Agent",
  "component_type": "agent",
  "description": "Routes incoming requests to specialized sub-agents based on intent classification"
}
```

### Model 2: AgentLink

Purpose: A directed causal link between two agent components. Wraps CausalLink internally for reuse of loop detection logic.

Fields:
- from_component: str (id of source AgentComponent)
- to_component: str (id of target AgentComponent)
- polarity: Polarity (same "+" / "-" enum as CausalLink)
- strength: Optional[float] (0.0-1.0, defaults to 1.0)
- label: str (optional, describes the interaction, e.g. "triggers retry", "provides context", max 200)
- link_type: Optional[Literal["data_flow", "control_flow", "feedback", "dependency"]] (categorizes the interaction)

Internal composition:
- AgentLink.to_causal_link() -> CausalLink: transforms to CausalLink(from_var=from_component, to_var=to_component, polarity=polarity, strength=strength) for reuse with score_reinforcing_loops

Validation rules:
- from_component and to_component must reference valid AgentComponent ids (cross-validated at tool input level, not model level)
- polarity required

Example payload:
```json
{
  "from_component": "agent-1",
  "to_component": "tool-1",
  "polarity": "+",
  "strength": 0.8,
  "label": "invokes tool on each request",
  "link_type": "control_flow"
}
```

### Model 3: MCPTool

Purpose: Represents an MCP tool in a tool stack diagram (used by ST-005)

Fields:
- tool_id: str (unique identifier, e.g. "score_reinforcing_loops")
- name: str (display name, min 1, max 200)
- server: str (MCP server name, e.g. "systems-thinking-cubelets")
- description: str (optional, what this tool does, max 500)
- inputs: List[str] (input parameter names, for dependency tracking)
- outputs: List[str] (output field names, for dependency tracking)
- estimated_latency_ms: Optional[int] (typical response time, for blast radius analysis)
- criticality: Optional[Literal["low", "medium", "high", "critical"]] (how important this tool is to the stack)

Validation rules:
- tool_id and name must be non-empty
- server must be non-empty

Example payload:
```json
{
  "tool_id": "score_reinforcing_loops",
  "name": "Score Reinforcing Loops",
  "server": "systems-thinking-cubelets",
  "description": "Graph traversal to find, classify, and score all feedback loops",
  "inputs": ["variables", "causal_links"],
  "outputs": ["loops", "variable_participation", "dominant_dynamic"],
  "estimated_latency_ms": 200,
  "criticality": "high"
}
```

### Model 4: ToolDependency

Purpose: A directed dependency between two MCP tools (used by ST-005)

Fields:
- from_tool: str (tool_id of the upstream tool)
- to_tool: str (tool_id of the downstream tool)
- dependency_type: Literal["required", "optional", "enhances"] (strength of dependency)
- data_fields: List[str] (which output fields from from_tool feed into to_tool inputs)
- description: str (optional, describes the dependency, max 300)

Validation rules:
- from_tool != to_tool (no self-dependencies)
- data_fields should be non-empty for "required" dependency_type

Example payload:
```json
{
  "from_tool": "create_causal_loop",
  "to_tool": "score_reinforcing_loops",
  "dependency_type": "required",
  "data_fields": ["variables", "causal_links"],
  "description": "CLD must be created before loops can be scored"
}
```

### Model 5: AutomationLayer

Purpose: Represents a layer in an automation system for debt detection (used by ST-006). Wraps FixRecord internally for reuse with detect_burden_shift.

Fields:
- layer_id: str (unique identifier)
- name: str (display name, e.g. "Human Review Queue", min 1, max 200)
- automation_type: Literal["quick_fix", "fundamental", "transition", "monitoring"] (classification of this automation layer)
- description: str (what this layer does, max 500)
- failure_mode: Optional[str] (how this layer fails, e.g. "Queue grows unbounded under load", max 300)
- tech_debt_indicator: Optional[str] (observable sign of debt, e.g. "Review backlog exceeds 48 hours", max 300)
- times_applied: int (how many times deployed/activated, default 1, min 1)
- side_effects: Optional[str] (known negative consequences, max 500)
- duration_active: Optional[str] (how long this layer has been running)

Internal composition:
- AutomationLayer.to_fix_record() -> FixRecord: transforms to FixRecord(label=name, description=description, times_applied=times_applied, is_symptomatic=(automation_type == "quick_fix"), side_effects=side_effects, duration_active=duration_active) for reuse with detect_burden_shift

Validation rules:
- Follow existing FixRecord validation patterns
- automation_type is the AI-specific extension: maps quick_fix -> symptomatic, fundamental -> fundamental, transition -> user-classified, monitoring -> symptomatic

Example payload:
```json
{
  "layer_id": "layer-1",
  "name": "Human Review Queue",
  "automation_type": "quick_fix",
  "description": "Route low-confidence LLM outputs to human reviewers",
  "failure_mode": "Queue grows unbounded when LLM confidence drops across the board",
  "tech_debt_indicator": "Review backlog exceeds 48 hours",
  "times_applied": 15,
  "side_effects": "Blocks parser improvement resources, creates reviewer fatigue",
  "duration_active": "6 months"
}
```

### Composition Map

Show how each new model transforms to existing models:

```
AgentLink.to_causal_link() --> CausalLink --> score_reinforcing_loops (existing)
AutomationLayer.to_fix_record() --> FixRecord --> detect_burden_shift (existing)
MCPTool + ToolDependency --> (new tool: analyze_tool_orchestration) --> compare_interventions (existing)
```

This composition pattern ensures all new MCP tools reuse proven analysis logic. The new tools handle domain-specific input validation and transformation; the existing tools handle the actual graph/scoring algorithms.
